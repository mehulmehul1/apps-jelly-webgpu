## PHASE 2: ADVANCED ELICITATION

### Method 1: Stakeholder Round Table
**Participants:** Marine Biologist, 3D Animator, Generative Artist

#### Marine Biologist Perspective:
**Key Questions:**
- What biological constraints must shape grammar respect?
- Which form features are taxonomically significant vs. individual variation?
- What developmental processes create observed forms?

**Insights:**
- **Taxonomic Significance:** Bell shape, tentacle arrangement, and internal anatomy are primary taxonomic characters
- **Developmental Constraints:** Forms emerge from embryonic tissue layers (ectoderm, endoderm, mesoglea)
- **Functional Trade-offs:** Transparency reduces structural support options, limiting form complexity
- **Symmetry Breaking:** Perfect radial symmetry is rare; most organisms have subtle asymmetries
- **Environmental Plasticity:** Same species shows different forms in different conditions

**Critical Requirements:**
1. Maintain developmental plausibility (forms must be biologically buildable)
2. Respect tissue property constraints (gelatinous materials behave differently)
3. Account for allometric growth (proportions change with size)
4. Include ontogenetic variation (juvenile vs. adult forms differ)

#### 3D Animator Perspective:
**Key Questions:**
- What forms are most animatable using WebGPU capabilities?
- Which form features create compelling motion?
- How should shape grammar support dynamic deformation?

**Insights:**
- **Animation Hierarchy:** Primary forms (bell) drive secondary (tentacles), tertiary (oral arms)
- **Deformation Zones:** Specific regions (bell margins, tentacle bases) are primary motion points
- **Rigging Challenges:** Soft-body simulation requires subdivision surface-friendly forms
- **Performance Optimization:** Fewer, larger forms animate better than many small ones
- **Visual Impact:** Silhouette variety matters more than internal complexity

**Critical Requirements:**
1. Generate animation-friendly topology (quad-based, regular subdivision)
2. Provide deformation weights per body region
3. Support procedural animation patterns (pulse, wave, undulation)
4. Enable LOD (Level of Detail) for performance scaling

#### Generative Artist Perspective:
**Key Questions:**
- What parameter spaces produce most aesthetic variety?
- How to balance randomness vs. recognizable forms?
- What creates satisfying visual discovery?

**Insights:**
- **Parameter Coupling:** Correlated parameters create organic coherence (size tentacles with bell)
- **Constrained Randomness:** Total randomness produces unnatural forms; guided exploration works better
- **Emergent Complexity:** Simple rules + layered modifiers create richness
- **Archetype Diversity:** Need distinct "families" of forms to maintain interest
- **Happy Accidents:** System should occasionally produce surprising but plausible forms

**Critical Requirements:**
1. Create parameter correlation rules (couple related features)
2. Provide archetype presets as starting points
3. Include "wildcard" parameters for occasional novelty
4. Support both guided and generative workflows

**Synthesis:**
Shape grammar must balance biological plausibility (biologist), animation performance (animator), and aesthetic variety (artist). Key tension: natural diversity vs. performance constraints. Resolution: hierarchical form generation with LOD support.

### Method 3: Architecture Decision Records
**Debate: Form System Architecture for Creature Generation**

#### Decision 1: Parametric vs. Procedural Generation
**Option A:** Parametric definition (define parameters, generate geometry)
**Option B:** Procedural generation (rules that iteratively build forms)

**Pros of A:**
- Predictable, reproducible results
- Easy to tune and debug
- Direct parameter mapping to UI controls
- Performance-friendly (known computation cost)

**Pros of B:**
- More organic, emergent forms
- Can generate unexpected variations
- Biologically plausible (development-like process)
- Infinite variety from simple rules

**Decision:** Hybrid approach - parametric base forms with procedural modifiers
- Use parameters for major features (bell shape, tentacle count)
- Use procedural generation for details (surface texture, tentacle branching)
- Rationale: Balances control (needed for animation) with variety (needed for aesthetics)

#### Decision 2: Hierarchical vs. Flat Form Structure
**Option A:** Hierarchical (body → regions → features → details)
**Option B:** Flat (all features generated simultaneously)

**Pros of A:**
- Natural dependency handling (tentacles need bell margin)
- Easier animation hierarchy
- Clear LOD implementation
- Maps to biological organization

**Pros of B:**
- Simpler implementation
- No dependency ordering issues
- Parallel generation possible

**Decision:** Hierarchical structure
- **Level 1:** Primary body (bell, tube, sphere)
- **Level 2:** Major features (tentacles, oral arms, comb rows)
- **Level 3:** Details (surface texture, bioluminescence patterns)
- **Level 4:** Dynamic modifiers (deformation weights, animation parameters)
- Rationale: Enables efficient LOD, natural animation, biological plausibility

#### Decision 3: Fixed Enum vs. Extensible Type System
**Option A:** Fixed enum (BodyPlan.Medusa, BodyPlan.CombJelly, etc.)
**Option B:** Extensible type system (add new types dynamically)

**Pros of A:**
- Type-safe, compile-time checking
- Known set of forms to optimize for
- Clear UI options
- Performance optimization possible

**Pros of B:**
- Unlimited form variety
- Easy to add new discoveries
- Future-proof
- Supports user creativity

**Decision:** Fixed enum with extensible parameters
- Keep BodyPlan enum (Medusa, CombJelly, Salp, Siphonophore)
- Add new enums only for fundamentally different architectures
- Use parameters within each type for variety
- Rationale: Balances type safety with extensibility, optimizes for common cases

#### Decision 4: Single Mesh vs. Multi-Piece Geometry
**Option A:** Generate single unified mesh per creature
**Option B:** Generate multiple mesh pieces (body + tentacles + details)

**Pros of A:**
- Simple rendering (single draw call)
- Easier material application
- Shared vertices for smooth blending

**Pros of B:**
- Independent animation of pieces
- LOD per piece
- Different materials per piece
- Performance optimization (don't update invisible pieces)

**Decision:** Multi-piece geometry with parent-child relationships
- Body mesh is parent
- Tentacle arrays are children
- Surface details are child meshes or shaders
- Rationale: Essential for natural animation (tentacles move independently), enables LOD

### Method 4: Pre-Mortem Analysis
**Scenario: Form System FAILS - What went wrong?**

#### Failure Scenario 1: "The Uncanny Valley of Sea Creatures"
**Problem:** Generated creatures look almost real but have subtle wrong details, creating discomfort rather than wonder.

**Root Causes:**
- Parameter combinations that violate biological constraints
- Symmetry that's too perfect (real organisms have variation)
- Proportions that don't match allometric scaling
- Surface features that don't follow developmental patterns

**Prevention Strategies:**
1. Implement "imperfection injection" (subtle asymmetry)
2. Add allometric scaling rules (proportions change with size)
3. Create constraint validation (flag biologically impossible forms)
4. Use reference-based parameter ranges (from real organisms)

#### Failure Scenario 2: "Animation Nightmare"
**Problem:** Forms look great but animate terribly (tentacles clip through body, bell collapses when pulsing).

**Root Causes:**
- Forms not designed with deformation in mind
- No consideration of animation topology
- Tentacle attachments in problematic locations
- No rigging/skinning information

**Prevention Strategies:**
1. Include animation-friendly topology in shape grammar
2. Generate weight maps with geometry
3. Constrain tentacle attachment zones to animatable areas
4. Test deformations during form generation

#### Failure Scenario 3: "Performance Meltdown"
**Problem:** Beautiful creatures but frame rate drops to 5 FPS.

**Root Causes:**
- Too many tentacles generated
- Excessive polygon counts from procedural details
- No LOD system
- Every creature has maximum complexity

**Prevention Strategies:**
1. Implement complexity budgets (max vertices, max tentacles)
2. Automatic LOD generation (simplify forms at distance)
3. Complexity scaling based on creature count
4. Performance metrics in generation phase

#### Failure Scenario 4: "Monotonous Ocean"
**Problem:** Despite many parameters, all creatures look the same.

**Root Causes:**
- Parameter ranges too narrow
- Too much coupling between parameters
- Lack of archetype diversity
- No consideration of visual silhouette variety

**Prevention Strategies:**
1. Broad parameter ranges (validated against real organisms)
2. Create distinct archetype families with unique characteristics
3. Emphasize silhouette diversity in generation
4. Include "wildcard" parameters for dramatic variations

#### Failure Scenario 5: "Extension Impossibility"
**Problem:** New creature type discovered but system can't support it without major rewrite.

**Root Causes:**
- Rigid coupling to existing body plans
- No extensibility in form generation
- Hardcoded assumptions about creature structure
- No plugin/modifier system

**Prevention Strategies:**
1. Design for extension from start (modifier pattern)
2. Keep body plans as compositions, not monoliths
3. Create plugin system for new form features
4. Document extension points and patterns

### Method 5: First Principles Analysis
**Question: What are the fundamental principles of creature form in this domain?**

#### Principle 1: **Form Follows Function (in Fluid Environment)**
- All forms are optimized for fluid dynamics (water is 800x denser than air)
- Drag minimization drives streamlined shapes
- Buoyancy management drives gas-filled structures
- Energy efficiency drives pulsing vs. drifting strategies

**Implication:** Shape grammar must produce hydrodynamically plausible forms
- Streamlined profiles for active swimmers
- Buoyant forms for passive drifters
- Surface area optimization for filter feeders

#### Principle 2: **Symmetry Enables Efficient Development**
- Radial symmetry allows identical repetition of structures
- Reduces genetic information needed
- Enables modular growth (add more of same)
- Simplifies neural organization (repeat same patterns)

**Implication:** Shape grammar should use symmetry as primary organizational principle
- Base forms should be radially symmetric by default
- Break symmetry intentionally with modifiers
- Support variable symmetry orders (4, 5, 6, 8, etc.)

#### Principle 3: **Material Properties Constrain Form**
- Gelatinous materials (mesoglea) have limited structural support
- Cannot support heavy appendages without reinforcement
- Transparency requires specific thickness limits
- Muscle tissue must be anchored to rigid structures

**Implication:** Shape grammar must respect material constraints
- Large tentacles need thicker bases or fewer tentacles
- Transparency affects form complexity (more opaque = more complex possible)
- Scale affects material properties (larger = more structural support needed)

#### Principle 4: **Allometry Rules Scaling**
- Proportions change with size (isometry is rare)
- Larger organisms have relatively smaller appendages
- Surface area to volume ratio affects form
- Developmental timing affects adult proportions

**Implication:** Shape grammar must implement allometric scaling rules
- Tentacle length scales differently than bell diameter
- Number of tentacles doesn't linearly scale with size
- Surface complexity relates to volume, not just size

#### Principle 5: **Modularity Enables Complexity**
- Colonial organisms build complex forms from simple units
- Repeated zooids create macroscopic structures
- Functional specialization allows division of labor
